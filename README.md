# NVP AI Agent Fault-Independence Experiment

A research-grade framework that re-imagines the Knight & Leveson (1986) N-version programming fault-independence experiment using AI coding agents (Cursor, Claude Code, Codex, Gemini CLI, OpenCode, and a mock adapter) instead of human programmers.

## Research Question

Do versions of the Launch Interceptor Program generated by different AI coding agents fail independently, or do they exhibit statistically significant coincident failures beyond what independence would predict?

- **H0**: Failures across agent-generated versions are independent
- **H1**: Failures are positively correlated (non-independent)

## Background

Knight & Leveson (1986) had 27 programmers independently implement the same specification. They found that coincident failures occurred far more often than expected under independence (z ≈ 100, p < 10⁻⁹). This framework applies the same methodology to AI agents.

## Benchmark

The **Launch Interceptor Program** from Knight & Leveson (1986). Agents read the **verbatim** markdown spec ([`benchmarks/launch_interceptor/spec_original.md`](benchmarks/launch_interceptor/spec_original.md); same text as [`KLspec.md`](KLspec.md) at repo root) and implement the program in one or more **target languages** configured under `languages:` (default `python`). Prompts and harnesses live in [`harnesses/`](harnesses/): Python (`decide.py`, in-process), Free Pascal (`lipdecide.pas` + JSON-lines driver; requires `fpc`), Rust (`decide.rs` + Cargo harness; requires `cargo`). The trusted oracle remains Python-only; all targets are compared to it. A Python-oriented developer summary lives in [`benchmarks/launch_interceptor/spec.md`](benchmarks/launch_interceptor/spec.md) and is **not** fed to agents.

## Quick Start

### Requirements

- Python 3.11+
- API keys (optional): `CURSOR_API_KEY`, `ANTHROPIC_API_KEY`, `CODEX_API_KEY`, `OPENROUTER_API_KEY` (OpenCode via OpenRouter), `GEMINI_API_KEY` (Gemini CLI; or `gemini_api_key` in YAML — do not commit). For Gemini from Cursor/IDE, copy [`.env.example`](.env.example) to `.env` and set `GEMINI_API_KEY` (loaded automatically).
- Agent CLIs (optional): `agent` (Cursor), `claude` (Claude Code), `codex` (OpenAI Codex), `gemini` (Google Gemini CLI), `opencode` (OpenCode)
- **Pascal / Rust targets** (optional): `fpc` (Free Pascal) for `languages` including `pascal` — only the core RTL units are required (the harness uses a small built-in JSON parser, not `fpjson`); `cargo`/`rustc` for `rust`

Without any API keys the mock agent runs automatically, validating the full pipeline **for Python only** (mock skips non-Python languages with `api_unavailable`).

### OpenCode with OpenRouter

Use a model id prefixed with `openrouter/` (this alone enables OpenRouter mode), or set `openrouter: true` under that agent’s `config`. Export your key:

```bash
export OPENROUTER_API_KEY="sk-or-..."
```

YAML examples:

```yaml
  - name: opencode
    runs: 1
    config:
      openrouter: true
      model: anthropic/claude-sonnet-4.5   # passed to CLI as openrouter/anthropic/...
      opencode_agent: build
```

Or set `model: openrouter/anthropic/claude-sonnet-4` directly. OpenRouter auth: use `opencode auth login` (same as TUI), and/or `OPENROUTER_API_KEY` in `.env`. Only set `openrouter_sandbox_config: true` if you need a generated sandbox `opencode.json` for CI (requires an API key in env). See [OpenCode providers](https://opencode.ai/docs/providers/).

### Install

The package name on PyPI is `pyyaml`; Python imports it as `yaml`. Install project dependencies from the repo root:

```bash
python3 -m venv .venv
.venv/bin/pip install -e .
```

(On Debian/Ubuntu, use a venv—system Python often blocks `pip install` without `--break-system-packages`.)

Alternatively: `.venv/bin/pip install -r requirements.txt` (no editable install), or only `pip install pyyaml` if everything else is already present.

### Run Oracle Tests

```bash
pytest oracle/ -v
```

### Run Pilot Experiment

In YAML, set **`languages`** (e.g. `[python]` or `[python, pascal, rust]`). Total versions per agent = `len(languages) × len(models) × runs`. Each agent can list **`config.models`** (list or single string). **`runs`** is the number of generations **per (model, language)**. Legacy **`config.model`** is treated as a one-element list.

`analyze_results` writes pooled `stats.json` plus **`stats_<lang>.json`** per language, and optional **cross-language** matched-pair analysis (`cross_language_pairwise.csv`, `cross_language_stats.json`, `--no-cross-language` to skip).

```bash
# 1. Generate agent versions (sandboxed per run)
python -m pipeline.generate_versions --config config/pilot.yaml --output results/pilot/versions/

# 2. Acceptance screening (50 random cases, all must pass)
python -m pipeline.run_acceptance --versions results/pilot/versions/ --output results/pilot/accepted.json

# 3. Shared randomized test campaign (1000 cases)
python -m pipeline.run_campaign --accepted results/pilot/accepted.json --config config/pilot.yaml --output results/pilot/campaign.csv

# 4. Statistical analysis + report
python -m analysis.analyze_results --campaign results/pilot/campaign.csv --output results/pilot/ --report report.md
```

Latest results in `results/main-spec-3/`: `campaign.csv` (gitignored; regenerate with step 3), `stats.json`, `summary_table.csv`, `failure_heatmap.png`.
Final report: `report.md`.

## Browse Implementations

Every per-agent/model/language version produced by `generate_versions` is archived under [`results/main/versions/`](results/main/versions/) as JSON (prompt + source + build/acceptance metadata). The [`docs/`](docs/) folder ships a static GitHub Pages site for browsing them interactively (filter by agent, model, language, build status; syntax-highlighted code viewer).

```bash
python3 docs/build.py                # regenerate docs/data/ from results/main/versions/
python3 -m http.server -d docs 8000  # preview locally at http://localhost:8000
```

To publish, enable **Settings → Pages → Deploy from branch** on your fork with folder `/docs`.

## Statistical Methodology

Follows Knight & Leveson exactly:

1. Per-version failure rate: `p_i = failures_i / T`
2. Expected simultaneous failures under H0: `P_m = 1 - P_0 - P_1`
3. K&L z-statistic: `z = (K - T·P_m) / sqrt(T·P_m·(1-P_m))`
4. Reject H0 at 99% confidence if `|z| > 2.576`

Secondary: pairwise binomial tests with Bonferroni correction, Pearson phi correlation.

## Key Differences from Original K&L

| Aspect | Knight & Leveson 1986 | This Framework |
|---|---|---|
| Programmers | 27 humans (undergrad/grad) | AI coding agents |
| Language | Pascal | Pascal / Python / Rust |
| Versions | 27 | 5 agents × 3 languages x N models |
| Test cases | 1,000,000 | 1,000,000 |
| Independence | Guaranteed by isolation | Operational |
| Primary confounder | Education similarity | Shared training data / model lineage |

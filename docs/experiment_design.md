# Experiment Design: NVP Fault-Independence with AI Coding Agents

## 1. Research Question

Do versions of the **Launch Interceptor Program** (LIP) generated by different AI coding agents fail independently on randomly drawn test inputs, or do they exhibit statistically significant coincident failures beyond what would be expected under an independence assumption?

This is a direct re-imagining of Knight & Leveson (1986), substituting AI coding agents for human programmers while preserving the core experimental methodology.

---

## 2. Hypotheses

**H0 (Null):** The failure events of independently generated agent versions are mutually independent. The probability of two or more versions failing simultaneously on a random test case equals the product of their individual failure probabilities.

Formally, for versions i and j with individual failure rates p_i and p_j:

```
P(fail_i ∩ fail_j) = p_i · p_j
```

And for the simultaneous-failure statistic K (number of test cases with 2+ failures):

```
E[K] = T · P_m   where P_m = 1 - P_0 - P_1
```

(see Section 7 for notation).

**H1 (Alternative):** Failures are positively correlated — coincident failures occur more often than independence predicts. This would indicate shared fault patterns caused by common training data, shared model lineage, common reasoning heuristics, or identical prompt text.

```
P(fail_i ∩ fail_j) > p_i · p_j
```

---

## 3. What "Independent Versions" Means in This Setting

In the original K&L experiment, independence was approached through:
- Different programmers with no interaction
- Different universities (different pedagogy)
- No sharing of code or design documents

In the AI-agent setting, **operational independence** is all we can claim:
- Each run is isolated: separate sandbox directory, no shared state
- The same prompt is used (identical text), but agents are invoked as separate processes
- No run sees the output of any other run before or during generation
- Different agent systems (Cursor, Claude Code, Codex) may use different underlying models

This is weaker than the human case. The confounders in Section 5 explain why.

## 4. The Benchmark: Launch Interceptor Program

The specification is taken verbatim from Knight & Leveson (1986). The task is to implement the `DECIDE` procedure:

**Input:**
- `NUMPOINTS` (2–100): number of planar radar data points
- `X`, `Y`: coordinate arrays (length NUMPOINTS)
- `PARAMETERS`: record with LENGTH1, RADIUS1, EPSILON, AREA1, Q_PTS, QUADS, DIST, N_PTS, K_PTS, A_PTS, B_PTS, C_PTS, D_PTS, E_PTS, F_PTS, G_PTS, LENGTH2, RADIUS2, AREA2
- `LCM` (15×15): Logical Connector Matrix with values {NOTUSED, ORR, ANDD}
- `PUM` diagonal: 15 booleans indicating whether each LIC should gate launch

**Output (241 comparison bits):**
- `CMV` (15 booleans): Conditions Met Vector — LIC 1 through LIC 15
- `PUM` off-diagonal (210 booleans): Preliminary Unlocking Matrix
- `FUV` (15 booleans): Final Unlocking Vector
- `LAUNCH` (1 boolean): final decision

**Known fault-attractor regions:**

| LIC | Geometric challenge | Common fault mode |
|-----|--------------------|--------------------|
| 2, 9, 14 | Smallest enclosing circle of 3 points | Wrong circumscribed circle algorithm; ignoring obtuse-triangle case |
| 3, 10 | Angle at vertex of three points | Coincident point check; atan2 vs dot-product method |
| 5 | Quadrant assignment | Boundary cases: (0,y)→Q1 not Q2; (x,0)→Q1 not Q4 |
| 7 | Distance from line segment | Confusion between point-to-line vs point-to-segment |
| 13, 14, 15 | Dual existence conditions | Missing the second sub-condition; wrong parameter reuse |
| All | REALCOMPARE semantics | Using == instead of 6-sig-digit tolerance |
| 8, 9, 10, 11, 12, 13, 14, 15 | "exactly K_PTS intervening" | Off-by-one: index i and i+K_PTS+1 vs i+K_PTS |

**Suitability for randomized testing:** The LIP has a large, continuous input domain (real coordinates in any range, continuous parameter space, discrete LCM). Random inputs exercise the full condition space. The acceptance filter ensures only working implementations enter the main campaign.

## 5. Key Confounders

These are threats to the validity of any independence claim in the AI-agent setting. They are documented here; the framework cannot eliminate them.

### 5.1 Shared Base Models

Multiple agent systems may run on the same or similar underlying language models (e.g., both Cursor and Claude Code may call Claude-class models). If so, their "independently generated" versions are really two draws from the same or related distributions, not independent implementations.

*Mitigation:* Record model names/versions in every `VersionRecord`. Stratify analysis by underlying model if detectable.

### 5.2 Shared Training Data

All large language models are trained on code from the same internet-scale corpus (GitHub, Stack Overflow, textbooks). A bug in widely-circulated reference implementations of, say, the smallest enclosing circle algorithm would be learned by all models and propagate to all agents.

*Mitigation:* None possible at this scale. Document as a primary confounder; note that this is analogous to "same university training" in K&L.

### 5.3 Identical Prompts

Unlike K&L, where each human received the same document but interpreted it independently, here all agents receive byte-for-byte identical prompts. Prompt text induces correlated reasoning: if the prompt mentions a specific algorithm or uses a particular framing, all agents may adopt that framing.

*Mitigation:* Use a single standard prompt derived directly from the spec with no algorithmic hints. Do not include code examples.

### 5.4 Deterministic Decoding

Some agent configurations (temperature=0) produce deterministic output. Multiple runs of the same agent with the same prompt at temperature=0 produce identical code, making intra-agent "independence" meaningless.

*Mitigation:* Run at non-zero temperature (temperature ≥ 0.2) for all agents. Record temperature in `generation_config`. Verify that intra-agent runs differ in source code before including multiple runs from the same agent.

### 5.5 Shared Repair Loops

If agents are allowed multiple turns (tool-use, self-correction), they may converge on similar solutions through similar repair strategies. A bug introduced in turn 1 may be fixed in turn 3 the same way by all agents because the error message pattern triggers the same repair heuristic.

*Mitigation:* Limit to single-turn generation where possible. Log all turns; if multi-turn, record the full conversation.

### 5.6 Prompt Similarity Across Agents

Because all agents receive the same task prompt, the "independence" of efforts is limited to: (a) different model weights, (b) different decoding stochasticity, (c) different tool use strategies. This is a much weaker form of independence than the K&L human case.

*Mitigation:* Treat this as a feature, not a bug — the experiment tests whether even weaker forms of diversity produce independent failures.

## 6. Threats to Validity

### Internal Validity

- **Oracle correctness**: If the reference implementation has bugs, failures will be systematically misclassified. Mitigation: extensive oracle test suite (`test_oracle.py`), including all spec examples.
- **Acceptance filter**: Acceptance cases are drawn from the same distribution as the campaign. If the filter uses a fixed seed different from the campaign seed, it is truly independent.
- **Sandbox leakage**: If agents share filesystem state between runs, independence is violated at the infrastructure level. Mitigation: each run uses a fresh temp directory.

### External Validity

- **Benchmark representativeness**: The LIP is a single problem. Results may not generalize to other domains (e.g., web services, numerical algorithms, string processing).
- **Agent representativeness**: The three agents available in March 2026 may be unrepresentative of the full space of AI coding agents.
- **Scale**: 1,000 test cases vs K&L's 1,000,000. This limits statistical power to detect small deviations from independence.

### Construct Validity

- **Failure definition**: A version "fails" on a test case if ANY of its 241 output bits differ from the oracle. This is the K&L definition and is conservative — partial correctness is still counted as failure.
- **"Independence"**: We test operational independence (process isolation + different agent systems), not mathematical independence. The latter cannot be guaranteed.

## 7. Statistical Methodology

Following Knight & Leveson (1986) exactly.

**Notation:**
- N: number of admitted versions
- T: number of campaign test cases
- f_i: number of test cases on which version i fails
- p_i = f_i / T: empirical failure rate of version i
- K: number of test cases on which 2 or more versions fail simultaneously

**Primary test — K&L z-statistic:**

Under H0:
```
P_0 = ∏_{i=1}^{N} (1 - p_i)
P_1 = Σ_{i=1}^{N} [ p_i · ∏_{j≠i} (1 - p_j) ]
P_m = 1 - P_0 - P_1
```

K is approximately normally distributed with mean `μ = T · P_m` and variance `σ² = T · P_m · (1 - P_m)`.

```
z = (K - μ) / σ
```

Reject H0 at 99% confidence if `|z| > 2.576`.

**Secondary — Pairwise analysis:**

For each pair (i, j):
- Observed co-failures: `O_ij = |{t : fail_i(t) AND fail_j(t)}|`
- Expected under H0: `E_ij = p_i · p_j · T`
- One-sided binomial test: `P(X ≥ O_ij | n=T, p=p_i·p_j)`
- Bonferroni correction: α_adj = 0.05 / C(N,2)
- Pearson phi coefficient: correlation between binary failure vectors

**Assumptions:**
1. Test cases are drawn i.i.d. from the input distribution.
2. Empirical failure rates are good estimates of true failure probabilities.
3. Under H0, failures of different versions on the same test case are independent Bernoulli events.
4. Normal approximation to Binomial is valid when T·P_m ≥ 5 (rule of thumb).

## 8. Reproducibility Plan

- Fixed seed: `seed: 42` in `config/pilot.yaml`
- All random draws use `numpy.random.default_rng(seed)` seeded from config
- Acceptance cases use seed `seed + 1000` (distinct from campaign seed)
- Every `VersionRecord` stores: agent name, model name/version, full prompt text, full source code, timestamp, generation config, acceptance result
- Results are deterministic given the same versions (oracle is deterministic)
- Docker/container recipe for full reproducibility is out of scope for this pilot but documented as a next step
- Optional **fault trace artifact** (`fault_log` in config or `--fault-log` on `run_campaign`): JSONL with the inputs for each failing (version, test) pair and a structured diff of CMV / PUM / FUV / LAUNCH mismatches (`fault_log_detail: summary` or `full`). The main `campaign.csv` stays slim for K&L statistics; the JSONL supports tracing failures to spec clauses.

---

## 8.1 Multi-language implementations and cross-language independence

The pipeline can request **Python**, **Free Pascal** (`fpc`), and **Rust** (`cargo`) via `languages:` in the experiment YAML. For each agent configuration, total generated versions = `len(languages) × len(models) × runs`. Version IDs include `__l_<language>__` (see `pipeline/naming.py`). The **trusted oracle remains Python-only** (`oracle/launch_interceptor/solution.py`); non-Python candidates are compiled once and exercised through a **JSON-lines harness** (`harnesses/`) so large campaigns stay tractable.

**Statistical evaluation:** `analysis/analyze_results.py` writes pooled `stats.json` (all versions), **per-language** outputs (`stats_<lang>.json`, heatmaps), and optional **matched cross-language** analysis: pairs of versions that share the same agent, model, and run index but differ in `language` are compared (same random test stream) to estimate **phi correlation** and excess coincident failures vs independence—useful for separating specification-level faults from language-runtime effects. Use `--no-cross-language` to skip that block.

## 9. Faithfulness to Knight & Leveson vs Adaptations

### Faithful elements

| K&L element | This framework |
|---|---|
| Same specification for all versions | Same verbatim K&L spec (`spec_original.md`) for all agents; language-specific delivery text in `harnesses/*_harness.py` (Python / Pascal unit / Rust function) |
| Independently produced versions | Separate sandboxed processes, no shared state |
| Acceptance filter before main campaign | 50-case acceptance suite; failed versions excluded |
| Single trusted oracle | `oracle/launch_interceptor/solution.py` |
| Shared randomized test set | Same 1,000 tests (from seed 42) run on every admitted version |
| K&L z-statistic for independence | Implemented exactly in `analysis/analyze_results.py` |

### Adaptations

| K&L design choice | Adaptation | Reason |
|---|---|---|
| Pascal DECIDE procedure | Configurable: Python `decide()`, Free Pascal `lipdecide` unit + harness globals, or Rust `decide()` + serde types | Default pilot uses Python; Pascal/Rust optional where toolchains are available |
| REALCOMPARE (6 sig. digits) | In-process `realcompare` in each submission, same semantics as K&L | Spec semantics preserved; see oracle `realcompare` |
| 27 human programmers | e.g. 3 agents × N models × R runs each (see `config/models` + `runs`) | Limited by API access and cost |
| 1,000,000 test cases | 1,000 (pilot) | Practical constraint; scalable by config |
| University students | State-of-the-art coding agents (2026) | The "subjects" are now AI systems |

## 10. Expected Outcomes by Scenario

| Scenario | z-statistic | Interpretation |
|---|---|---|
| All agents correct | z ≈ 0 | Framework works; no failures to analyze |
| Independent failures | |z| < 2.576 | Fail to reject H0; consistent with independence |
| Correlated failures | z >> 2.576 | Reject H0; evidence of shared fault patterns |
| Mock-only run | z designed to show both cases | Framework validation only |

## 11. Limitations and Next Steps

**Limitations of this pilot:**
- Small N (up to 9 versions) limits statistical power
- Single benchmark; not a multi-task study
- Agent non-determinism means "same agent" runs may differ across days
- API costs and rate limits constrain scale

**Next steps for a full study:**
- Multiple benchmarks from different domains
- More runs per agent (N ≥ 20 per system)
- Larger campaign (100,000+ test cases)
- Use different prompt formulations to measure prompt-sensitivity
- Add agents: GitHub Copilot, additional CLIs, etc. (framework includes `gemini` and `opencode` adapters.)
- Full reproducibility packaging (Docker + frozen model versions)
- Fault localization: which LICs drive coincident failures?
